# ADR-010: 分步执行中的滚动上下文聚合 (Rolling Context Aggregation in Step-by-Step Execution)

## Status
Accepted

## Context
分步执行机制（Step-by-Step）虽然解决了复杂任务的失控问题，但也引入了新的风险：AI 可能仅关注当前的子任务片段，而忽略了前置步骤已经生成的具体内容细节。这种“管中窥豹”的模式容易导致最终拼凑出的长文档出现逻辑断层、术语冲突以及内容厚度大幅缩水（往往不足 2000 汉字）的问题。

## Decision
1. **引入“步前读回”规约**：在 `meta-step-runner` 中强制要求 AI 在执行每个涉及文档或代码生成的步骤前，显式执行 `read_file` 以重新读取当前已生成的全量产物。
2. **物理缓存化策略**：将物理文件（而非内存缓存）视为 AI 的外部长时记忆，确保后续步骤始终是在“已有厚度”的基础上进行深化。
3. **强制实时落地 (Write-Through)**：每一任务步的产出必须立即写入文件，严禁保留内存草稿，确保读回的内容始终是最新的。
4. **注入终期质量审计步**：在所有功能任务完成后，强制执行“终期质量审计”，负责全局通读、润色及细节补全，确保最终产出的丰满度优于单一输出 baseline。

## Consequences
- **好处**：彻底解决了分步执行导致的内容缩水问题；保证了长文档的逻辑连贯性与深度；突破了 LLM 单次回复的 Token 输出限制。
- **约束**：增加了每回合的 Token 消耗（因为需要读回前文）；执行过程需要 AI 具备更强的全局统筹能力。
